# 注塑企业论文知识库 Agent 项目设计文档

## 1. 项目一句话定位

**注塑企业论文知识库 Agent** 是一个面向注塑成型企业的企业级 RAG + Agent 应用，基于注塑成型相关学术论文构建专业知识库，支持论文知识问答、缺陷诊断、工艺参数建议、论文方法对比、结构化知识卡抽取，并可作为求职面试中的工业 AI 项目展示。

简单来说，本项目不是普通 FAQ 问答系统，而是把大量注塑成型论文中的专业知识、实验结论、工艺规律、缺陷分析方法和算法模型整理成一个可检索、可问答、可调用工具、可评测的企业级智能助手。

---

## 2. 企业真实业务痛点

注塑成型企业在实际生产和研发过程中，常常会遇到以下问题。

### 2.1 论文知识难以被工程人员直接使用

注塑成型相关论文数量很多，内容涉及材料、模具、工艺参数、缺陷机理、数值模拟、质量预测、机器学习优化等方向。  
但是论文通常存在以下问题：

- 篇幅长，阅读成本高；
- 英文论文多，新员工和一线工程师理解困难；
- 论文结论分散在摘要、方法、实验和结论中；
- 同一类问题可能分布在很多篇论文里；
- 工程人员很难快速判断某篇论文的方法是否适合企业场景。

因此，企业虽然拥有大量论文资料，但这些资料往往没有真正转化为可复用的工程知识。

### 2.2 注塑缺陷诊断依赖经验，难以标准化

注塑生产中常见缺陷包括：

- 短射；
- 飞边；
- 缩痕；
- 气泡；
- 银纹；
- 熔接痕；
- 翘曲；
- 烧焦；
- 尺寸不稳定；
- 重量波动；
- 透明件透过率下降。

这些缺陷往往和多个因素有关，例如熔体温度、模具温度、注射速度、保压压力、保压时间、冷却时间、材料干燥、模具排气、浇口设计等。

实际企业中，缺陷诊断经常依赖老师傅经验。不同工程师给出的判断可能不一致，新员工也很难快速学习。  
因此，企业需要一个能够结合论文知识、缺陷机理和工艺经验的辅助诊断工具。

### 2.3 工艺参数调整缺少知识支撑

在注塑过程中，工艺参数之间存在复杂耦合关系。比如：

- 提高熔体温度可能改善流动性，但也可能导致材料降解；
- 提高注射速度可能减少短射，但也可能增加飞边或烧焦风险；
- 提高保压压力可能改善缩痕，但也可能导致内应力增加；
- 延长冷却时间可能改善尺寸稳定性，但会降低生产效率。

企业工程师在调参时，不仅需要知道“应该调什么参数”，还需要知道“为什么这样调”“有哪些风险”“论文中有没有类似案例”。

普通大模型容易直接给出笼统建议，而本项目希望通过 RAG 检索论文依据，让参数建议更加可解释、可追溯、可审核。

### 2.4 研发人员需要快速比较论文方法

研发人员、算法工程师和工艺优化人员经常需要回答：

- 哪些论文用了机器学习做注塑质量预测？
- 哪些论文使用了神经网络、随机森林、支持向量机、遗传算法、贝叶斯优化？
- 某篇论文的数据来源是什么？
- 输入变量和输出质量指标是什么？
- 这篇论文的方法能不能迁移到企业现有数据？
- 不同论文方法之间有什么优缺点？

如果人工逐篇阅读，效率很低。  
因此，本项目希望支持论文方法对比和结构化知识卡抽取，把论文转化为可比较、可复用的知识资产。

### 2.5 企业知识沉淀困难

企业内部经常存在以下问题：

- 新员工不知道从哪里学习注塑工艺；
- 老员工经验难以沉淀；
- 论文、报告、实验记录、缺陷案例分散存储；
- 项目复盘后没有形成结构化知识；
- 算法项目做完后，知识没有进入可复用系统。

本项目希望把论文知识库作为起点，逐步扩展到企业内部工艺案例、质量问题记录、实验数据和算法模型说明，最终形成注塑企业的智能知识中台。

## 3. 目标用户

本项目面向注塑成型企业中的多类用户，不同用户关注点不同。

### 3.1 工艺工程师

工艺工程师主要关注生产现场问题，例如：

- 某种缺陷可能由哪些参数导致；
- 当前缺陷应该优先调整哪些工艺参数；
- 调参时有哪些副作用；
- 有没有论文或案例支持这个判断；
- 如何向质量部门或主管解释调参依据。

对工艺工程师来说，本系统的价值是：  
**把论文中的工艺规律转化为可操作、可解释的调参建议。**

### 3.2 质量工程师

质量工程师主要关注产品质量稳定性，例如：

- 某类缺陷的形成原因；
- 哪些过程变量会影响质量；
- 如何建立质量问题分析报告；
- 如何追溯缺陷和工艺参数之间的关系；
- 哪些论文研究过质量预测或质量控制。

对质量工程师来说，本系统的价值是：  
**辅助质量问题分析，提高缺陷诊断和报告撰写效率。**

### 3.3 研发人员

研发人员主要关注新材料、新工艺、新模具和新方法，例如：

- 某种材料在注塑中的研究进展；
- 某类工艺优化方法的优缺点；
- 哪些论文适合参考；
- 某篇论文的实验设计是否可靠；
- 如何把论文方法转化为企业研发方案。

对研发人员来说，本系统的价值是：  
**快速获取论文方法、实验结论和可迁移思路。**

### 3.4 新员工

新员工通常缺少注塑经验，面对大量专业概念容易不理解，例如：

- 保压是什么意思；
- 短射和飞边分别是什么；
- 为什么模具温度会影响翘曲；
- 为什么材料干燥会影响银纹；
- 注塑周期包括哪些阶段。

对新员工来说，本系统的价值是：  
**提供通俗易懂的注塑知识解释，帮助快速入门。**

### 3.5 算法工程师

算法工程师主要关注如何把论文知识和企业数据结合，例如：

- 哪些论文用了机器学习模型；
- 输入特征和预测目标是什么；
- 是否有可复现的数据结构；
- 哪些方法适合做质量预测、缺陷分类、参数优化；
- 如何设计 RAG、Agent、评测集和工具调用流程。

对算法工程师来说，本系统的价值是：  
**提供从论文知识到工业 AI 应用落地的完整技术链路。**

## 4. 核心功能列表

### 4.1 论文知识问答

用户可以直接向系统提问，例如：

- 注塑成型中保压压力对产品重量有什么影响？
- 哪些论文研究了注塑件翘曲预测？
- 熔体温度和模具温度分别会影响哪些质量指标？
- 透明件透过率下降可能和哪些工艺参数有关？
- 注塑质量预测常用哪些机器学习方法？

系统通过检索论文片段，生成带有依据的回答，而不是凭空回答。

核心目标：

- 回答要基于论文内容；
- 尽量给出来源；
- 对零基础用户要解释清楚；
- 对专业用户要保留技术细节；
- 不确定时明确说明无法判断。

### 4.2 注塑缺陷诊断

用户可以输入缺陷现象，例如：

- 产品出现短射；
- 产品边缘有飞边；
- 表面出现银纹；
- 产品有缩痕；
- 透明件透过率不稳定；
- 产品重量波动较大。

系统根据论文知识和预设诊断逻辑，输出：

- 可能原因；
- 相关工艺参数；
- 优先排查顺序；
- 建议调整方向；
- 调整风险；
- 需要人工确认的信息；
- 可参考的论文依据。

该功能不是替代工程师决策，而是辅助工程师形成更系统的分析思路。

### 4.3 工艺参数建议

系统可以根据缺陷或质量目标，给出初步参数调整建议。例如：

- 短射时可以考虑提高熔体温度、提高注射速度、增加注射压力，但要注意飞边和烧焦风险；
- 缩痕时可以考虑增加保压压力或保压时间，但要注意内应力和尺寸变化；
- 翘曲时需要综合考虑模具温度、冷却时间、保压阶段和材料收缩差异。

输出建议时必须包含：

- 调整方向；
- 可能改善的原因；
- 可能带来的副作用；
- 不建议自动执行的说明；
- 建议人工验证的实验方案。

### 4.4 论文方法对比

系统可以对多篇论文进行结构化比较，例如：

- 研究问题是什么；
- 使用了什么模型；
- 输入变量有哪些；
- 输出质量指标是什么；
- 数据来自仿真、实验还是真实生产；
- 方法优点是什么；
- 方法局限是什么；
- 是否适合企业落地。

示例问题：

- 对比几篇注塑质量预测论文中的机器学习方法。
- 哪些方法适合小样本注塑数据？
- 神经网络方法和传统机器学习方法在注塑质量预测中有什么区别？
- 某篇论文的方法能否用于企业的工艺参数推荐？

### 4.5 结构化知识卡抽取

系统可以从论文中抽取结构化知识卡，便于后续检索、对比和复用。

知识卡可以包括：

- 论文标题；
- 作者；
- 年份；
- 研究对象；
- 材料；
- 工艺场景；
- 数据来源；
- 输入变量；
- 输出质量指标；
- 使用方法；
- 主要结论；
- 工程启发；
- 局限性；
- 是否适合企业应用；
- 可用于哪些问答场景。

知识卡的作用是把长篇论文转化为短小、清晰、可检索的知识单元。

### 4.6 论文摘要和通俗解释

系统可以把复杂论文内容解释成不同层次：

- 一句话总结；
- 面向新员工的通俗解释；
- 面向工艺工程师的应用解释；
- 面向算法工程师的方法解释；
- 面试展示版解释。

这样可以让同一篇论文服务于不同用户。

### 4.7 Agent 工具调用

本项目不只是普通 RAG 问答，还设计 Agent workflow，让模型可以根据任务自动选择工具。

可设计的工具包括：

- 论文检索工具；
- metadata 查询工具；
- 缺陷诊断规则工具；
- 参数建议模板工具；
- 知识卡抽取工具；
- 方法对比工具；
- 引用来源检查工具；
- 回答安全检查工具；
- 人工评测题调用工具。

Agent 的作用是把复杂任务拆成多个步骤，而不是一次性让大模型直接回答。

### 4.8 评测与反馈

为了证明项目不是简单 Demo，需要设计评测体系。

评测内容包括：

- 检索结果是否相关；
- 回答是否基于论文；
- 是否出现幻觉；
- 缺陷诊断是否合理；
- 参数建议是否安全；
- 引用来源是否正确；
- 回答是否适合零基础用户；
- 回答是否适合面试展示。

评测方式包括：

- 人工构造问题集；
- 标准答案或参考答案；
- 检索命中率评估；
- 回答质量打分；
- 人工专家复核；
- 错误案例分析。

## 5. 明确边界：模型自动做什么，人工必须介入什么

企业级 AI 项目必须明确边界。  
本项目不是让模型完全替代工程师，而是让模型辅助工程师更快、更系统地完成知识检索、分析和决策准备。

### 5.1 可以让模型自动完成的事情

以下任务适合由模型自动完成：

#### 论文检索

模型可以根据用户问题，从向量库和关键词索引中检索相关论文片段。

例如：

- 查找“翘曲”和“模具温度”相关论文；
- 查找“质量预测”和“神经网络”相关论文；
- 查找“保压压力”和“产品重量”相关结论。

#### 论文内容总结

模型可以把论文片段总结成更容易理解的语言。

例如：

- 总结某篇论文研究了什么；
- 提炼论文主要方法；
- 解释论文结论；
- 说明论文对企业有什么启发。

#### 初步缺陷原因分析

模型可以基于论文知识和规则模板，列出可能原因。

例如：

- 短射可能与熔体温度低、注射压力不足、流动性差、排气不良有关；
- 飞边可能与锁模力不足、注射压力过高、模具磨损有关；
- 银纹可能与材料含水率、熔体温度、剪切过强有关。

#### 初步工艺参数建议

模型可以给出方向性建议，而不是直接给出最终生产参数。

例如：

- 可以建议“优先检查材料干燥状态”；
- 可以建议“尝试小幅提高保压压力”；
- 可以建议“结合实际设备限制进行试验验证”。

#### 论文方法对比

模型可以自动对比多篇论文的方法、输入、输出、优缺点和适用场景。

#### 知识卡抽取

模型可以从论文中抽取结构化知识卡，作为后续知识库的一部分。

#### 回答格式整理

模型可以根据用户身份调整输出格式。

例如：

- 给新员工：更通俗；
- 给工程师：更重视调参逻辑；
- 给算法工程师：更重视模型结构和数据输入输出；
- 给面试官：更重视项目完整性和工程落地。

### 5.2 必须人工介入的事情

以下任务不能完全交给模型自动完成，必须由人工介入。

#### 最终工艺参数决策

模型不能直接决定生产线最终参数。

原因是：

- 论文场景和企业现场可能不同；
- 材料、设备、模具、环境差异很大；
- 参数调整可能导致质量风险；
- 生产参数涉及成本、安全和交付责任。

模型只能提供建议，最终必须由工艺工程师确认。

#### 真实生产异常的最终判定

模型可以辅助分析缺陷原因，但不能替代现场工程师判断。

现场还需要检查：

- 模具状态；
- 设备状态；
- 原料批次；
- 干燥条件；
- 操作记录；
- 传感器数据；
- 产品实物缺陷形貌。

#### 论文结论的工程适用性判断

论文结论不一定能直接迁移到企业。

人工需要判断：

- 论文材料是否相同；
- 产品结构是否类似；
- 设备是否相似；
- 工艺窗口是否一致；
- 数据规模是否足够；
- 方法是否可复现。

#### 知识卡质量审核

模型自动抽取的知识卡可能存在遗漏或误解，因此需要人工抽检和审核。

尤其是以下字段需要重点检查：

- 输入变量；
- 输出指标；
- 实验条件；
- 主要结论；
- 方法局限；
- 工程启发。

#### 高风险建议审核

涉及安全、设备保护、大批量生产、客户交付的建议，必须人工审核。

例如：

- 大幅提高注射压力；
- 大幅提高熔体温度；
- 改变模具结构；
- 改变材料配方；
- 直接批量生产验证。

### 5.3 项目中的安全原则

本项目的基本原则是：

> 模型负责检索、总结、分析和建议；人负责验证、决策和承担生产责任。

所有涉及实际生产调整的回答，都应该包含以下提醒：

- 建议仅作为参考；
- 需要结合材料、模具、设备和现场条件；
- 需要小范围试验验证；
- 不能直接自动下发到生产设备；
- 高风险调整必须由工程师确认。

## 6. 数据来源

本项目的数据来源主要包括以下几类。

### 6.1 注塑论文 PDF

这是知识库的核心数据来源。

论文内容可以包括：

- 注塑成型工艺；
- 注塑缺陷分析；
- 工艺参数优化；
- 产品质量预测；
- 机器学习建模；
- 数值模拟；
- 模具设计；
- 材料性能；
- 过程监测；
- 智能制造。

PDF 是主要原始资料，但 PDF 不能直接用于 RAG，需要经过解析、清洗、切分和索引。

### 6.2 论文 metadata

metadata 是论文的结构化信息，包括：

- 标题；
- 作者；
- 年份；
- 期刊或会议；
- DOI；
- 关键词；
- 摘要；
- 文件路径；
- 研究方向标签；
- 材料类型；
- 方法类型；
- 是否与缺陷诊断相关；
- 是否与质量预测相关；
- 是否与参数优化相关。

metadata 的作用是帮助系统进行过滤、排序、展示和管理。

例如，当用户问“近几年有哪些注塑质量预测论文”时，metadata 可以帮助按年份筛选。

## 7. 技术路线

本项目的技术路线可以理解为一条完整流水线：

> PDF 文献 → 文本解析 → 数据清洗 → chunk 切分 → embedding → 向量库 → 混合检索 → rerank → RAG 回答 → Agent 工具调用 → 安全约束 → 评测 → Streamlit 展示

下面按模块说明。

### 7.1 PDF 解析

PDF 解析的目标是把论文 PDF 转换成可处理的文本。

需要处理的问题包括：

- PDF 中的标题、摘要、正文、表格、图注、参考文献；
- 多栏排版；
- 页眉页脚；
- 公式；
- 表格；
- 图片说明；
- OCR 识别错误；
- 英文断行；
- 特殊符号乱码。

可选工具包括：

- PyMuPDF；
- pdfplumber；
- unstructured；
- Nougat；
- GROBID；
- MinerU；
- OCR 工具。

对于零基础项目，可以先用简单 PDF 解析工具跑通流程；后续再逐步提高解析质量。

### 7.2 文本清洗

PDF 解析后得到的文本通常很乱，需要清洗。

清洗内容包括：

- 删除页眉页脚；
- 删除重复标题；
- 合并断行；
- 修复英文单词断裂；
- 删除参考文献中无关内容；
- 保留摘要、方法、实验、结论等关键部分；
- 统一空格和标点；
- 过滤过短或无意义段落。

清洗的目标不是把文本变得“好看”，而是让后续检索和问答更准确。

### 7.3 Chunk 切分

chunk 是 RAG 中的基本检索单元。  
如果 chunk 太长，检索不精准；如果 chunk 太短，上下文不完整。

本项目可以使用多种切分策略：

#### 按固定长度切分

例如每 500 到 800 个 token 切一段，并保留一定 overlap。

优点是简单。  
缺点是可能把完整语义切断。

#### 按论文结构切分

根据摘要、引言、方法、实验、结果、结论等章节切分。

优点是语义更完整。  
缺点是需要更好的 PDF 结构解析。

#### 按知识类型切分

把不同类型内容拆成不同 chunk，例如：

- 缺陷原因；
- 工艺参数影响；
- 实验方法；
- 模型结构；
- 数据集；
- 结论；
- 局限性。

这种方式更适合工业知识库，但实现难度更高。

初版项目可以先采用“结构优先 + 固定长度兜底”的策略。

### 7.4 Embedding

embedding 的作用是把文本转成向量，使系统能够根据语义相似度检索内容。

例如用户问：

> 为什么保压压力会影响产品重量？

系统不一定只检索包含完全相同关键词的句子，而是可以检索到语义相关内容，例如：

- packing pressure；
- holding pressure；
- part weight；
- shrinkage；
- dimensional stability。

embedding 模型可以选择：

- 通用中文 embedding 模型；
- 中英文混合 embedding 模型；
- 面向论文或技术文档的 embedding 模型；
- 本地开源 embedding 模型；
- API embedding 模型。

由于注塑论文中大量内容是英文，因此需要重点考虑中英文混合检索能力。

### 7.5 向量库

向量库用于存储 chunk 的向量和 metadata。

可选方案包括：

- Chroma；
- FAISS；
- Milvus；
- Qdrant；
- Weaviate；
- Elasticsearch 向量检索。

初版项目可以使用 Chroma 或 FAISS，因为部署简单。  
如果后续做企业级部署，可以考虑 Milvus、Qdrant 或 Elasticsearch。

每个 chunk 需要保存：

- chunk 文本；
- 所属论文；
- 页码；
- 章节；
- 年份；
- 关键词；
- 研究方向；
- chunk 类型；
- 向量表示。

这样回答时可以追溯来源。

### 7.6 Hybrid Retrieval

只使用向量检索可能会漏掉一些专业词。  
例如：

- PC；
- PMMA；
- warpage；
- short shot；
- weld line；
- holding pressure；
- injection speed。

这些词有时需要关键词精确匹配。  
因此，本项目应采用 hybrid retrieval，也就是混合检索。

混合检索包括：

- 向量检索：找到语义相近内容；
- 关键词检索：找到精确术语匹配内容；
- metadata 过滤：按年份、材料、方法、缺陷类型筛选；
- 结果融合：把不同检索结果合并排序。

这样可以提高专业问答的稳定性。

### 7.7 Rerank

初步检索出的结果不一定都相关，因此需要 rerank。

rerank 的作用是对候选 chunk 重新排序，把最相关的内容放在前面。

例如用户问：

> 注塑中熔体温度对透明件透过率有什么影响？

初步检索可能找到很多关于熔体温度的内容，但其中只有一部分真正和透明件、光学质量、透过率有关。  
rerank 可以帮助系统优先使用最相关的论文片段。

### 7.8 RAG 回答生成

RAG 的核心流程是：

1. 用户提出问题；
2. 系统检索相关论文片段；
3. 对检索结果进行重排序；
4. 把最相关内容提供给大模型；
5. 大模型基于论文内容生成回答；
6. 回答中给出依据和不确定性说明。

本项目中的 RAG 回答需要遵守以下原则：

- 优先基于检索内容回答；
- 不知道就说明不知道；
- 不编造论文结论；
- 参数建议必须说明风险；
- 缺陷诊断必须说明需要人工验证；
- 面向零基础用户时要解释术语；
- 面向工程师时要给出排查顺序。

### 7.9 Agent Workflow

Agent workflow 是本项目的亮点之一。  
普通 RAG 只是“检索 + 回答”，而 Agent 可以根据任务类型自动规划步骤。

例如用户问：

> 产品出现银纹，可能是什么原因？有没有相关论文依据？应该怎么调整参数？

Agent 可以拆成以下步骤：

1. 识别任务类型：缺陷诊断 + 参数建议 + 论文依据；
2. 调用缺陷诊断工具，查找银纹可能原因；
3. 调用论文检索工具，检索 silver streak、splay mark、moisture、melt temperature 等相关内容；
4. 调用参数建议工具，生成可操作建议；
5. 调用安全检查工具，检查是否存在高风险建议；
6. 生成最终回答，包括原因、建议、风险和论文依据。

这种设计可以体现项目不是简单调用大模型，而是具备任务编排能力。

### 7.10 Guardrails

guardrails 是安全约束，用来防止模型输出不可靠或危险内容。

本项目需要的 guardrails 包括：

#### 幻觉控制

如果检索结果中没有依据，模型不能编造论文结论。

回答应说明：

- 当前知识库没有检索到足够依据；
- 只能给出通用经验判断；
- 建议补充相关论文或企业数据。

#### 参数建议安全约束

模型不能直接给出高风险绝对指令，例如：

- “立即把注射压力提高到某个值”；
- “直接批量生产”；
- “不用验证”。

正确做法是：

- 给出方向性建议；
- 建议小范围试验；
- 说明副作用；
- 要求工程师确认。

#### 来源可追溯

涉及论文结论时，应尽量给出来源信息，例如论文标题、页码、章节或 chunk 来源。

#### 用户身份适配

同一个问题，不同用户需要不同解释深度。  
例如新员工需要通俗解释，算法工程师需要模型和数据细节。

### 7.11 Evaluation

评测是企业级项目和普通 Demo 的重要区别。

本项目可以设计以下评测指标：

#### 检索评测

评估系统能否找到正确论文片段。

指标包括：

- Top-k 命中率；
- 相关性得分；
- 是否命中目标论文；
- 是否命中正确章节。

#### 回答质量评测

评估模型回答是否可靠。

维度包括：

- 是否回答了问题；
- 是否基于论文；
- 是否存在幻觉；
- 是否逻辑清晰；
- 是否适合目标用户；
- 是否说明不确定性。

#### 缺陷诊断评测

评估缺陷诊断是否合理。

维度包括：

- 原因是否全面；
- 排查顺序是否合理；
- 参数建议是否安全；
- 是否说明副作用；
- 是否要求人工验证。

#### 知识卡评测

评估知识卡抽取是否准确。

维度包括：

- 标题是否正确；
- 方法是否正确；
- 输入输出是否正确；
- 结论是否准确；
- 局限性是否遗漏；
- 是否适合后续检索。

### 7.12 Streamlit 展示

Streamlit 用于快速搭建项目 Demo 页面，方便面试展示。

页面可以包括：

#### 首页

展示项目名称、项目定位、技术路线和核心能力。

#### 论文问答页面

用户输入论文相关问题，系统返回基于 RAG 的回答。

#### 缺陷诊断页面

用户选择或输入缺陷类型，系统输出可能原因、参数建议和风险提醒。

#### 论文方法对比页面

用户选择多篇论文，系统生成对比表。

#### 知识卡抽取页面

用户上传或选择论文，系统生成结构化知识卡。

#### 评测页面

展示评测问题、系统回答、人工评分和错误案例。

#### 项目说明页面

展示技术架构、数据流程、边界说明和 GitHub 公开说明。

Streamlit 的优点是开发快、展示直观，适合求职面试和项目演示。

### 8.1 为什么做这个项目

普通 FAQ 是人工提前写好的问答，问题和答案通常比较固定。  
我的知识库主要来自学术论文，内容更长、更复杂、更专业。  
所以项目重点不是简单问答，而是解决论文解析、知识抽取、专业检索、方法对比和工程应用转化的问题。

5. Agent 层：设计任务分类、工具调用和工作流；

```text
.env
*.key
*.pem

data/raw_pdfs/
data/private/
data/full_text/
data/vector_db/
data/embeddings/
data/internal_cases/

outputs/private/
logs/
cache/
__pycache__/
```

这样可以避免把论文原文、向量库、密钥和企业数据误传到 GitHub。

### 9.5 GitHub 展示策略

GitHub 不需要公开全部真实数据。  
更推荐的展示方式是：

- 公开完整代码框架；
- 公开项目设计文档；
- 公开少量可运行样例；
- 公开 Demo 截图；
- 公开评测方法；
- 不公开完整论文库；
- 不公开企业数据；
- 不公开密钥和私有配置。

面试时可以解释：

> 出于论文版权和企业数据安全考虑，我没有公开完整论文库和向量库，而是公开了代码框架、样例数据、知识卡格式、评测问题和 Demo 说明。真实数据可以在本地运行时构建索引。

## 10. 项目总结

“注塑企业论文知识库 Agent” 的核心价值，是把注塑成型论文中的专业知识转化为企业可用的智能系统。

它不是简单的聊天机器人，而是围绕注塑企业真实业务痛点，构建了从论文数据治理到 RAG 检索问答、从缺陷诊断到工艺参数建议、从知识卡抽取到 Agent 工具调用、从安全边界到评测部署的完整项目。

对于求职面试来说，这个项目能够体现：

- 机械与注塑成型背景；
- 工业知识库建设能力；
- RAG 和 Agent 应用能力；
- 工业 AI 落地意识；
- 数据治理、评测和部署能力；
- 对企业真实业务边界的理解。

最终，这个项目可以作为一个非常适合展示“机械制造 + AI 应用 + 工程化落地”的综合型项目。
